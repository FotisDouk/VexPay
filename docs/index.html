VexPay documentation

Accept cryptocurrency on your own server, non-custodially. VexPay derives a receive address per invoice, watches the chain, and confirms payments automatically — funds go straight to your wallet. VexPay never holds your keys or your money.

Non-custodial

You supply an xpub/address/view-key. No private keys on the server.

Self-hosted

One Go binary or Docker container. SQLite by default.

Any language

A clean REST API and a flagship PHP SDK.

Real commerce

Fiat pricing, locked rates, signed webhooks, QR codes.

This site is a single self-contained HTML file — no build step, no external services. Open it in a browser or serve the docs/ folder with any static file server.

Quickstart

Run the gateway (requires Go 1.21+, or use Docker below):

git clone https://github.com/vexarnetwork/vexpay
cd vexpay
go run ./cmd/vexpay

In development a sandbox API key is generated and printed to the log. Create an invoice, then simulate paying it — no real coins needed:

KEY="vpk_test_..."   # from the startup log

# Create a sandbox invoice
curl -s -X POST http://localhost:8080/v1/invoices \
  -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" \
  -d '{"chain":"mock","asset":"tBTC","amount":"0.0025"}'

# Simulate the payment (sandbox only)
curl -s -X POST http://localhost:8080/v1/sandbox/pay/<invoice-id> \
  -H "Authorization: Bearer $KEY"

The invoice transitions to paid and a signed webhook fires (if configured).

Self-hosting

Docker (recommended)

cd deploy
cp vexpay.env.example vexpay.env   # then edit vexpay.env
docker compose up -d

The SQLite database is persisted in a named volume, so invoices survive restarts.

Bare binary

go build -o vexpay ./cmd/vexpay
VEXPAY_ENV=production \
VEXPAY_PUBLIC_URL=https://pay.example.com \
VEXPAY_ADMIN_SESSION_SECRET=$(openssl rand -hex 32) \
./vexpay

In production, VEXPAY_PUBLIC_URL must use HTTPS and VEXPAY_ADMIN_SESSION_SECRET is required. Put VexPay behind a TLS-terminating reverse proxy, or supply certificates.

Wallet setup

VexPay is watch-only. How you connect a wallet depends on the chain — but in every case you provide public material only, never a private key.

Bitcoin

Provide an account extended public key (xpub/zpub) in the invoice's wallet.xpub. VexPay derives a fresh native-SegWit address per invoice at m/0/index. Funds arrive directly in your wallet; because each invoice gets its own address, matching is exact.

{
  "chain": "bitcoin",
  "asset": "BTC",
  "amount": "0.005",
  "wallet": { "xpub": "zpub6r..." }
}
Roadmap: account-based chains (ETH/EVM, Tron) will use a single receive address plus a unique per-invoice amount, and Monero will use a view key with a subaddress per invoice.

API reference

Authenticate every request with Authorization: Bearer <api-key> (or the X-API-Key header). The full machine-readable spec is api/openapi.yaml.

EndpointDescription
POST/v1/invoicesCreate an invoice (crypto- or fiat-priced)
GET/v1/invoices/{id}Fetch an invoice and its live status
GET/v1/invoices/{id}/qrPNG QR of the payment URI (?size)
GET/v1/invoicesList invoices (?limit, ?offset)
POST/v1/sandbox/pay/{id}Simulate a payment (sandbox keys only)

Fiat-priced invoice

Omit asset/amount and give a fiat price; VexPay fetches and locks the rate, rounding up so you are never underpaid.

curl -X POST http://localhost:8080/v1/invoices \
  -H "Authorization: Bearer $KEY" -H "Content-Type: application/json" \
  -d '{"chain":"bitcoin","fiat_currency":"EUR","fiat_amount":"25","wallet":{"xpub":"zpub..."}}'

Invoice status values

pending · underpaid · confirming · paid · overpaid · expired

Webhooks

When an invoice changes state, VexPay POSTs a signed JSON event to your VEXPAY_WEBHOOK_URL. Delivery retries with backoff until it gets a 2xx.

Each request carries a VexPay-Signature: t=<unix>,v1=<hex> header. Verify it by computing HMAC-SHA256(secret, "<t>." + rawBody) and comparing in constant time, then reject timestamps outside your tolerance to prevent replay.

// PHP (via the SDK)
$event = \VexPay\Webhook::constructEvent($payload, $header, $secret);
if ($event->invoice->isPaid()) { /* fulfil order */ }
Always verify the signature before trusting the payload, and acknowledge quickly (2xx) so retries stop.

Sandbox & testing

The mock chain lets you exercise the entire flow — create, receive, confirm, settle, webhook — without spending real coins. Sandbox keys start with vpk_test_ and only they may call the simulator.

# Fully pay (default): confirmations = required, amount = full
curl -X POST .../v1/sandbox/pay/<id> -H "Authorization: Bearer $KEY"

# Partial / custom
curl -X POST .../v1/sandbox/pay/<id> -H "Authorization: Bearer $KEY" \
  -H "Content-Type: application/json" -d '{"amount":"0.001","confirmations":0}'

Security model

  • Non-custodial by design. No private keys and no funds live on the server, so the most valuable target simply isn't there.
  • Fail closed. If a rate or confirmation is uncertain, an invoice is never marked paid.
  • Exact money. Amounts are integers in the smallest unit — no floating point in balances or comparisons.
  • Authenticated webhooks. HMAC-SHA256 signatures with timestamped replay protection.
  • Hashed API keys. Stored as SHA-256, compared in constant time.

Found a vulnerability? See SECURITY.md for private disclosure.

SDKs

The REST API is language-agnostic; the PHP SDK is the flagship client.

composer require vexpay/sdk
$vexpay  = new \VexPay\Client('https://pay.example.com', 'vpk_live_...');
$invoice = $vexpay->createInvoice([
    'chain' => 'bitcoin', 'asset' => 'BTC', 'amount' => '0.005',
    'wallet' => ['xpub' => 'zpub...'],
]);
echo $invoice->paymentUri;

See sdks/php/README.md for the full SDK guide.
